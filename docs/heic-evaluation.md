# HEIC/HEIF 技术评估

> 评估日期：2026-07-17
> 当前结论：默认版本暂不接入 HEIC/HEIF，不引入 CGO 或系统动态库。

## 1. 目标与约束

Feather-ImgBed 当前坚持：

- `CGO_ENABLED=0`。
- Linux `amd64`、`arm64` 使用同一套构建流程。
- 默认镜像保持单 Go 二进制、无运行时图像编解码系统库。
- 原图永远保留，浏览器不支持的格式通过派生预览展示。
- 所有输入继续接受文件大小、像素数量、真实 MIME 和解码错误检查。

HEIF 是容器，HEIC 通常表示容器中的 HEVC 编码图片。不能只检查扩展名或单一 magic 后就假定其中一定是可解码的普通静态图片；同一容器还可能含多个图像、序列、辅助图、深度图、10 位/HDR 色彩和变换信息。

## 2. 候选实现

### 2.1 libheif + HEVC 解码器

[libheif](https://github.com/strukturag/libheif) 功能最完整，支持 HEIC/AVIF、多图、缩略图、方向变换、ICC/NCLX 色彩信息以及 EXIF/XMP。HEIC 解码通常还依赖 libde265 或 FFmpeg 插件。

优点：

- 格式覆盖和元数据支持成熟。
- 可以正确处理容器变换、嵌入缩略图和色彩配置。
- 有清晰的 C API，其他语言绑定较多。

代价：

- Go 绑定通常需要 CGO，或要求镜像携带动态库及其编解码插件。
- `amd64`、`arm64` 都要独立构建、更新和扫描本地依赖。
- 镜像体积、CVE 响应面和构建时间显著增加。
- 插件组合不同会造成“能识别容器但无法解码其中编码”的运行时差异。

这条路径只适合未来的可选增强镜像，不应进入当前默认镜像。

### 2.2 CGo-free WASM / 动态库混合方案

[gen2brain/heic](https://github.com/gen2brain/heic) 提供不依赖 CGO 的 Go API：优先尝试通过 purego 加载系统 libheif，缺失时回退到内嵌 WASM；也可使用 `nodynamic` 强制 WASM。

优点：

- 可以保持 `CGO_ENABLED=0`。
- WASM 回退让 `amd64`、`arm64` 更容易共用构建流程。
- 不要求默认环境一定安装 libheif。

风险：

- “不使用 CGO”不等于没有本地/运行时复杂性；动态加载路径仍依赖平台库，WASM 路径会增加二进制体积和运行时内存。
- 两条执行路径必须产生一致的方向、色彩和错误语义。
- 超大 iPhone 照片解码时，原始 YUV、RGBA 输出、WASM 线性内存和 JPEG 预览可能同时存在，峰值内存需要实测。
- 动态库路径存在平台差异；项目 issue 已记录过 Windows 动态模式问题，说明必须按目标平台做实际验证。

如果后续必须在默认镜像实验 HEIC，这一方案比 CGO 更接近现有发布约束，但仍不能在没有基准与安全测试时直接启用。

## 3. 必须完成的验证矩阵

### 3.1 构建与镜像

| 项目 | `linux/amd64` | `linux/arm64` | 通过条件 |
| --- | --- | --- | --- |
| `CGO_ENABLED=0 go build` | 必测 | 必测 | 两者均无需系统头文件或链接器 |
| Docker 多架构构建 | 必测 | 必测 | 当前流水线不增加架构特例 |
| 冷启动 | 必测 | 必测 | 无缺少动态库或插件错误 |
| 镜像/二进制增量 | 记录 | 记录 | 增量可接受且有发布说明 |
| 漏洞扫描 | 必测 | 必测 | 无无法维护的本地编解码依赖 |

### 3.2 正确性语料

至少准备并保留可再分发的回归样本：

- iPhone 8 位与 10 位 HEIC。
- 竖拍、横拍、镜像及裁剪变换。
- 带 EXIF Orientation，同时又带 HEIF `irot`/`imir` 的文件。
- Display P3、sRGB、ICC 和 NCLX 色彩描述。
- HDR/高位深样本；第一版即使只输出 SDR JPEG，也必须定义色调映射策略。
- 带 alpha、嵌入缩略图、多图和 Live Photo 相关容器。
- 1200 万、2400 万、4800 万像素及更大的照片。
- 截断文件、错误 box 长度、超大尺寸声明、递归/异常 item 引用等恶意样本。

libheif 的能力说明明确包含色彩配置、变换与 EXIF/XMP；这些不能在转换时静默忽略。现实中的高分辨率 HEIC 也可能是分块、10 位、带方向和 NCLX 配置的组合，[相关性能案例](https://github.com/strukturag/libheif/issues/1787)展示了 6336×9504 分块图像及丰富元数据，因此只用几张普通 iPhone 样本不足以验收。

### 3.3 资源与安全

对每个架构记录：

- 解码耗时、预览 JPEG 编码耗时和总上传耗时。
- 峰值 RSS、Go heap、WASM 线性内存。
- 并发 1、2、3 个大图时的峰值。
- 像素限制在读取完整像素前能否生效。
- 请求取消后，解码是否能及时停止并释放内存。
- 文件大小与像素数比值极端时是否可能造成内存放大。

建议第一版为 HEIC 使用比普通格式更保守的并发信号量，并继续复用全局最大像素限制。若解码库无法在读取配置阶段可靠给出尺寸，必须先解析容器元数据或在隔离进程中处理，不能先无界解码再检查。

## 4. 建议的产品语义

技术验证通过后的第一版也应坚持非破坏处理：

1. 原始 HEIC 文件保存为原图，MIME 与扩展名按真实内容确定。
2. 生成静态 `kind=preview` 的 JPEG 或 PNG 派生。
3. 图库使用预览；详情明确标注原文件与预览。
4. 复制链接时由用户选择原文件或预览，不静默把 `url` 改成 JPEG。
5. 多图/动画容器第一版只使用主图，并在元数据中记录“已选择主图”；不能假装完整保留了展示语义。
6. EXIF 默认不复制到公开预览，避免位置等隐私信息；方向和色彩渲染结果必须正确。
7. 解码失败不删除已经安全保存的原文件，但必须返回可理解的“预览不可用”状态。

浏览器上传与 API 客户端必须进入同一 `ingestImage` 流水线，不能让某一入口绕过 HEIC 像素、大小或派生失败策略。

## 5. 结论

当前不实现 HEIC/HEIF 主线支持，原因不是文件无法保存，而是“可靠展示”同时涉及 HEVC 解码、容器变换、高位深/HDR、色彩配置、EXIF 隐私和较高的峰值内存。

后续推荐顺序：

1. 在独立实验分支比较 `gen2brain/heic` 的 `nodynamic` WASM 路径与可选 libheif 增强镜像。
2. 完成上述双架构语料、内存和色彩验证。
3. 先发布可选增强镜像或显式实验开关。
4. 只有当默认纯 Go 路径在体积、内存和正确性上均达标时，才考虑加入默认镜像。

在此之前，默认构建继续只接受 JPEG、PNG、GIF 和 WebP，并保持纯 Go、单二进制与多架构发布能力。
